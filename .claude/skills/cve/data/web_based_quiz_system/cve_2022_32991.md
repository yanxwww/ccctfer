    - 访问目标Web Based Quiz System实例并注册普通账户，然后登录，拿到有效的会话Cookie
    - 进入答题流程，定位到带有q=quiz与step=2的页面，请求中的eid参数就是注入点
    - 先用最小化探测确认注入是否成立，例如通过单引号、布尔判断或报错差异观察页面响应是否异常

漏洞利用方法
    - 一条完整的利用流程如下。第一步，访问目标 Web Based Quiz System 实例并注册普通账户，然后登录，拿到有效的会话 Cookie。第二步，进入答题流程，定位到带有 q=quiz 与 step=2 的页面，请求中的 eid 参数就是注入点。第三步，先用最小化探测确认注入是否成立，例如通过单引号、布尔判断或报错差异观察页面响应是否异常。第四步，在确认漏洞后，对 eid 参数执行数据库枚举，逐步获取当前数据库、表名、列名以及可读记录；若后端支持多语句执行，还可以利用堆叠查询把额外 SQL 写入数据库。第五步，在靶场化场景中，攻击者会借助该注入点把预置的账号数据写入或读出，再从数据库中提取下一阶段系统的管理员凭据，用于进入后续目标服务。这个漏洞的核心价值是未授权数据库读取；而在题目场景里，它还被进一步包装成攻击链的第一跳，用来为后续 RCE 提供凭据材料。
    - 可复现的示范性 PoC 可以分为手工验证和自动化验证两类。手工验证时，先以已登录状态访问类似 http://target:9002/welcome.php?q=quiz&step=2&eid=1 的正常请求，然后把 eid 改成会改变查询逻辑的输入，例如在原值后附加单引号、布尔条件、联合查询片段或多语句片段，若页面内容、题目加载结果或报错行为出现明显差异，则说明参数未被安全处理。自动化验证时，可以直接对这个已登录请求使用 sqlmap，例如：sqlmap -u "http://target:9002/welcome.php?q=quiz&step=2&eid=1" --cookie="PHPSESSID=<session>" -p eid --dbms=mysql --level=5 --risk=3 --batch --dump。若需要验证数据库写入能力，可在确认支持堆叠查询后，把针对 users 表的插入语句作为后续载荷执行，再回头查询该表是否出现新记录。由于给定材料没有附带独立的 exploit 脚本源码，上述 PoC 以基于真实注入点的请求构造和 sqlmap 调用为主。
